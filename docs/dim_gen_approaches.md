# 维度泛化方案对比

## 方案一：符号化方案（当前实现）

```
符号化 → 具体化 → 泛化
   ↓         ↓        ↓
 分析相似性  问LLM   生成变体
```

**步骤**：
1. 分析 Tensor 相似性，提取公共维度，符号化为 S0、S1
2. 问 LLM 某种符号模式经常取哪些值
3. 代入生成变体

**缺点**：流程复杂，需要符号化过程

## 方案二：直接改模型方案（新思路）

```
问LLM要配置 → 改meta → 跑模型 → 根据日志修复model.py
```

**步骤**：
1. **问 LLM 要 9 组常用输入配置**
   - 输入：模型类型、输入 tensor 形状
   - 输出：9 组具体的维度值组合

2. **改 meta**
   - 修改 `weight_meta.py` 中的维度值
   - 修改 `input_meta.py` 中的维度值

3. **跑模型**
   - 运行 `run_model`
   - 收集错误日志

4. **根据日志修复 model.py**
   - 分析维度不匹配的错误
   - 修改 model.py 中的硬编码维度

**优点**：
- 不需要符号化过程
- 直接针对运行时修复
- 更实用、更直接

**示例**：

```bash
# Step 1: 问 LLM
# Q: 这是一个 LLM 模型，输入形状 [1, 128, 4096]，请给出 9 组常用维度配置
# A: seq_len 可以取 64, 128, 256, 512, 1024...

# Step 2: 改 meta
# 将 weight_meta.py 中的维度值改为新值

# Step 3: 跑模型
python3.10 -m graph_net.torch.run_model --model-path <sg_path>

# Step 4: 根据日志修复
# 错误: shape '[1, 128, 128]' is invalid for input of size 8192
# 修复: 将 model.py 中的 128 改为正确的维度值
```

## 方案选择

| 场景 | 推荐方案 |
|------|---------|
| 需要通用框架支持 | 方案一（符号化） |
| 快速处理特定模型 | 方案二（直接改） |
| 批量自动处理 | 方案一 |
| 手动精调 | 方案二 |

## 方案二实现计划

1. `ops.py ask-llm-config` - 问 LLM 要 9 组配置
2. `ops.py apply-config` - 应用配置到 meta 文件
3. `ops.py run-and-fix` - 跑模型并自动修复 model.py
